1. 개요 및 목적
================================================================================

1.1 배경
--------
- Local LLM (Ollama 기반)은 고객사 기밀 정보 보호에 최적
- 그러나 학습 데이터의 시점 한계로 최신 AI 기술/트렌드 정보 부족
- 최신 AI 프레임워크, 클라우드 서비스, 규제 변화 등 실시간 정보 필요

1.2 목적
--------
- 고객사 민감 정보의 외부 유출 완전 차단
- 최신 AI 기술 동향 및 Best Practice 활용
- 컨설팅 품질과 보안의 균형 달성

1.3 핵심 원칙
-------------
[원칙 1] 고객 식별 정보는 절대 온라인으로 전송하지 않음
[원칙 2] 기술적 질의만 온라인 LLM에 요청
[원칙 3] 모든 온라인 통신은 암호화 및 로깅
[원칙 4] 민감도 분류 기반 자동 필터링

3.2 구성 요소별 역할
--------------------

[Query Router - 질의 분류기]
목적: 입력 질의를 분석하여 Local/Online LLM으로 라우팅
분류 기준:
  - 고객 데이터 포함 여부
  - 최신 정보 필요 여부
  - 질의 유형 (기술/비즈니스/일반)

[Sanitizer - 비식별화 처리기]
목적: 온라인 전송 전 모든 민감 정보 제거/대체
처리 내용:
  - 고유명사 → 일반 명사 대체
  - 수치 → 범위 또는 비율로 변환
  - 식별 가능 조합 정보 분리

[Online LLM API Gateway]
목적: 온라인 LLM API 호출 관리 및 보안
기능:
  - API 키 관리
  - 요청/응답 로깅
  - Rate Limiting
  - 비용 관리

[Response Merger - 응답 통합기]
목적: Local/Online 응답을 통합하여 최종 결과 생성
기능:
  - 응답 일관성 검증
  - 고객 컨텍스트 재결합
  - 품질 평가

5. 익명화/비식별화 기법
================================================================================

5.1 텍스트 익명화 전략
----------------------

[전략 1: 대체 (Substitution)]
목적: 식별 정보를 일반적인 대체어로 교체
예시:
  Before: "삼성전자의 AI 전환 프로젝트"
  After:  "[대기업 A]의 AI 전환 프로젝트"

[전략 2: 일반화 (Generalization)]
목적: 구체적 값을 범위/카테고리로 변환
예시:
  Before: "연 매출 5,230억원"
  After:  "연 매출 5,000억원 이상 대기업"

[전략 3: 억제 (Suppression)]
목적: 불필요한 식별 정보 완전 제거
예시:
  Before: "대표이사 김철수가 승인한 AI 예산 50억원"
  After:  "경영진이 승인한 AI 예산"

[전략 4: 노이즈 추가 (Perturbation)]
목적: 수치에 무작위 변동 추가
예시:
  Before: "직원 수 1,247명"
  After:  "직원 수 약 1,200명 수준"

5.2 컨텍스트 분리 기법
----------------------

원본 질의:
"삼성전자(연매출 300조)의 스마트팩토리 AI 도입 시
 예상 ROI를 분석해주세요. 현재 데이터사이언티스트 50명 보유."

분리 후 Online 전송:
"대규모 제조기업의 스마트팩토리 AI 도입 시
 일반적인 ROI 계산 방법론과 업계 벤치마크를 알려주세요."

분리된 Local 처리:
- 회사명: 삼성전자 → [Company_ID: C001]
- 매출: 300조 → [Financial_Data: F001]
- 인력: 50명 → [HR_Data: H001]

6.3 하이브리드 처리 예시
------------------------

[시나리오: AI 플랫폼 추천]

사용자 입력:
"우리 회사(연매출 800억, 제조업, 데이터사이언티스트 5명)에 적합한
 MLOps 플랫폼을 추천해주세요."

처리 과정:

Step 1: Query Router 분류
  - 민감 정보 탐지: 연매출 800억, 제조업, 인원 5명
  - 질의 유형: 기술 추천 (Online 활용 가능)
  - 결정: 하이브리드 처리

Step 2: Sanitizer 처리
  온라인 전송용 질의:
  "중견 제조기업(매출 500-1000억 규모)에 적합한 MLOps 플랫폼을
   추천해주세요. AI 팀 규모는 10명 미만이며, 비용 효율성이 중요합니다.
   온프레미스와 클라우드 하이브리드 환경을 고려합니다."

  Local 보관 정보:
  - exact_revenue: 800억
  - exact_headcount: 5명
  - company_id: internal_ref_001

Step 3: 온라인 LLM 응답
  "중견 제조기업을 위한 MLOps 플랫폼 추천:
   1. MLflow (오픈소스, 저비용)
   2. Kubeflow (확장성, K8s 기반)
   3. AWS SageMaker (관리형, 하이브리드 지원)
   ..."

Step 4: Local LLM 보완 처리
  "해당 고객사의 구체적인 상황(800억 매출, 5명 팀)을 고려할 때,
   초기에는 MLflow로 시작하여 점진적으로 확장하는 것을 권장합니다.
   현재 인력 규모로는 관리형 서비스 부담이 클 수 있습니다..."

Step 5: Response Merger 통합
  최종 응답 생성 (Online 기술 정보 + Local 맞춤 분석)

7.2 핵심 컴포넌트 상세
----------------------

[A] Query Router & Classifier
-----------------------------
기능: 입력 질의 분석 및 라우팅 결정
구현:
  - NER (Named Entity Recognition) 기반 민감 정보 탐지
  - 키워드 기반 필터링 (블랙리스트/화이트리스트)
  - 질의 유형 분류 (기술/비즈니스/일반)

분류 로직:
  1. 민감 정보 스캔
  2. 질의 의도 파악
  3. 라우팅 결정:
     - 민감 정보 有 + 최신 정보 불필요 → Local Only
     - 민감 정보 有 + 최신 정보 필요 → 하이브리드 (익명화 후)
     - 민감 정보 無 + 최신 정보 필요 → Online 허용
     - 민감 정보 無 + 최신 정보 불필요 → Local (효율성)

[B] Data Sanitizer
------------------
기능: 온라인 전송 전 민감 정보 제거/대체
구현:
  - 정규표현식 기반 패턴 매칭
  - 사전 기반 고유명사 탐지
  - 문맥 기반 민감도 평가 (Local LLM 활용)

처리 파이프라인:
  1. 토큰화
  2. 엔티티 인식
  3. 민감도 분류
  4. 대체/제거 처리
  5. 일관성 검증

[C] Audit Logger
----------------
기능: 모든 온라인 통신 기록 및 감사
기록 항목:
  - 타임스탬프
  - 원본 질의 해시
  - 익명화된 질의
  - 사용된 Online LLM
  - 응답 요약
  - 처리 소요 시간

[D] API Gateway (Outbound)
--------------------------
기능: 외부 API 호출 관리
보안 기능:
  - API 키 암호화 저장 (Vault)
  - 요청/응답 TLS 1.3 암호화
  - IP 화이트리스팅
  - Rate Limiting
  - 비용 제한 (일/월 한도)

[E] Cache Layer
---------------
기능: 중복 질의 최적화 및 응답 캐싱
구현:
  - 질의 임베딩 기반 유사도 매칭
  - TTL 기반 캐시 만료 (기술 정보: 7일, 가격 정보: 1일)
  - 캐시 적중 시 온라인 호출 생략

[Query Router 핵심 로직]

```python
class QueryRouter:
    """질의 라우팅 결정기"""

    def __init__(self):
        self.sensitive_patterns = self._load_patterns()
        self.online_keywords = ['최신', '2024', '트렌드', '비교', '벤치마크']

    def route(self, query: str) -> RoutingDecision:
        sensitivity = self._check_sensitivity(query)
        needs_online = self._needs_online_info(query)

        if sensitivity >= SensitivityLevel.CONFIDENTIAL:
            if needs_online:
                return RoutingDecision.HYBRID  # 익명화 후 온라인
            return RoutingDecision.LOCAL_ONLY

        if needs_online:
            return RoutingDecision.ONLINE_ALLOWED

        return RoutingDecision.LOCAL_PREFERRED

    def _check_sensitivity(self, query: str) -> SensitivityLevel:
        for pattern, level in self.sensitive_patterns.items():
            if re.search(pattern, query):
                return level
        return SensitivityLevel.PUBLIC
```

[Data Sanitizer 핵심 로직]

```python
class DataSanitizer:
    """데이터 비식별화 처리기"""

    def sanitize(self, query: str, context: dict) -> SanitizedQuery:
        # 1. 엔티티 추출
        entities = self._extract_entities(query)

        # 2. 대체 맵 생성
        replacement_map = {}
        for entity in entities:
            if entity.type == 'COMPANY':
                replacement_map[entity.text] = '[기업 A]'
            elif entity.type == 'MONEY':
                replacement_map[entity.text] = self._generalize_amount(entity.text)
            elif entity.type == 'PERSON':
                replacement_map[entity.text] = '[담당자]'

        # 3. 대체 적용
        sanitized = query
        for original, replacement in replacement_map.items():
            sanitized = sanitized.replace(original, replacement)

        # 4. 원본 매핑 저장 (Local)
        return SanitizedQuery(
            original=query,
            sanitized=sanitized,
            mapping=replacement_map
        )
```
